## Context

Whisper models (particularly those fine-tuned on French subtitle data) embed a deterministic watermark phrase when given very short or silent audio clips. The most common variants observed:

- French: `"Sous-titres réalisés par Whisper"`, `"Sous-titres réalisées par"`
- English: `"Subtitles generated by Whisper"`
- Variations with/without trailing spaces, punctuation, or extra whitespace

This occurs in `AudioEngine.transcribe()` when the audio is too short for meaningful speech detection (e.g., user pressed and released Fn without speaking).

## Goals / Non-Goals

**Goals:**
- Strip all known Whisper credit/watermark prefixes from transcription output
- Handle both French and English variants
- Make the cleaning function robust to variations in whitespace and punctuation
- Add comprehensive unit tests

**Non-Goals:**
- Fixing the root cause in Whisper (out of scope)
- Adding UI configuration for which credits to strip
- Supporting languages beyond French and English

## Decisions

### Decision 1: Clean at the AudioEngine level
**Choice**: Apply credit stripping in `AudioEngine.transcribe()` before returning text.
**Rationale**: This keeps the cleaning logic close to the source of the problem. The `run_transcription()` method in `Engine` calls `transcribe()` and passes the result to the text injector, so cleaning at the `transcribe()` level ensures all callers benefit.

**Alternatives considered**:
- Clean in `Engine.run_transcription()` — would work but couples the engine to Whisper-specific knowledge
- Clean in `TextInjector.inject()` — too late in the pipeline, hides the issue

### Decision 2: Use a regex-based prefix matcher
**Choice**: Use `re.sub()` with an anchored pattern to match and remove the credit prefix.
**Rationale**: Regex handles whitespace variations and is simple to maintain. An anchored pattern (`^`) ensures we only strip from the beginning, not mid-text.

**Alternatives considered**:
- List-based check — brittle, doesn't handle whitespace variations
- Post-processing with string splitting — less precise, more error-prone

### Decision 3: Support both French and English credits
**Choice**: Include both language variants in the regex pattern.
**Rationale**: Users may have different language models loaded, and the credit may appear regardless of the configured language setting.

## Risks / Trade-offs

- **Risk**: Over-stripping legitimate text that happens to start with similar words.
  **Mitigation**: The pattern is specific enough (includes "Whisper" and the exact credit phrasing) that false positives are unlikely.

- **Risk**: New Whisper variants introduce new credit strings over time.
  **Mitigation**: The regex allows easy extension — just add new alternatives to the pattern.

- **Trade-off**: Adds a small overhead to every transcription.
  **Mitigation**: Regex on a short string is negligible cost (< 0.1ms).
